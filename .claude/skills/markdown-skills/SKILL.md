## 概要

このスキルは Markdown ファイルの作成と編集における一貫した記述ルールを提供します。ドキュメントの可読性と保守性を向上させ、プロジェクト全体で統一された形式を保ちます。

Markdown ドキュメントは技術仕様書、設計書、README など様々な用途で使用され、明確な構造と統一された記述規則により、チーム全体での情報共有を効率化します。

## このスキルの使い方

このスキルは以下の状況で自動的に有効化されます:

1. **新規 Markdown ドキュメント作成時**
   - ユーザーが `.md` ファイルの作成を依頼したとき
   - テンプレートベースのドキュメント生成時

2. **既存 Markdown ドキュメント編集時**
   - 既存の `.md` ファイルの修正や更新を行うとき
   - ドキュメントのリファクタリング時

3. **ドキュメントレビュー時**
   - Markdown ファイルの品質チェック
   - 記述規則への準拠確認

**使用方法:**
- 特別なコマンドは不要です
- Markdown ファイルに関する作業を依頼すると自動的にこのスキルの規則が適用されます
- より詳細なガイドラインは [resources/](./resources/) ディレクトリを参照してください

## ワークフロー判断

### 新規ドキュメント作成

新規 Markdown ドキュメントを作成する際は、以下の手順に従ってください:

1. **目的の明確化**
   - ドキュメントの種類を特定（設計書、README、技術仕様書など）
   - 対象読者を想定
   - 必要な情報の範囲を定義

2. **テンプレートの選択**
   - 設計書: `docs/design.md.sample` をベースに作成
   - 一般ドキュメント: このスキルの記述規則に従って作成
   - プロジェクト固有のテンプレートがあれば優先使用

3. **見出し構造の設計**
   - H1 でドキュメントタイトルを定義（1つのみ）
   - H2-H3 で論理的な階層構造を構築
   - H4 以降は使用せず、リストのネストで表現
   - 目次の追加を検討（長文の場合）

4. **内容の組み立て**
   - MECE（漏れなく重複なく）原則で情報を整理
   - 関連する内容をセクションでグループ化
   - 論理的な順序（時系列、重要度順など）で配置

5. **記述規則の適用**
   - リストはアスタリスク（*）とタブを使用
   - コードブロックに言語指定を追加
   - 英語と日本語の間に半角スペース

6. **レビューと調整**
   - 構造の妥当性を確認
   - 記述規則への準拠をチェック
   - 可読性を検証

### 既存ドキュメント編集

既存の Markdown ドキュメントを編集する際は、以下の手順に従ってください:

1. **現状の把握**
   - ドキュメント全体を読み、構造と内容を理解
   - 既存の記述スタイルを確認
   - 修正が必要な箇所を特定

2. **既存の記述規則に準拠**
   - 見出しレベルの維持（H1-H3）
   - リスト形式の統一（アスタリスク使用）
   - スペーシングルールの遵守
   - 既存のトーンとスタイルに合わせる

3. **内容の整理**
   - MECE 原則での再構成
   - 重複情報の統合
   - 論理的な順序での配置
   - 古い情報の更新

4. **コードブロックの保持**
   - 既存のコードは作業証跡として保持
   - 改変が必要な場合は新しいコードブロックを追加
   - コメントで変更理由を説明

5. **一貫性の確保**
   - 用語の統一
   - 書式の統一
   - リンクの有効性確認

6. **変更の検証**
   - 修正箇所の妥当性確認
   - 全体の整合性チェック
   - 可読性の維持確認

## 記述規則

### 見出し

* H1 はドキュメントのタイトルとして利用します
* H2 から H3 までを使用します
* H4 以降は基本的に使用しません
* 見出しに連番（1. や 1.1 など）を使用しません
* 見出しを細かく切らず、リストのネストで階層構造を表現します

実装例:

修正前:
```markdown
# システム設計
#### 詳細なコンポーネント設計
##### フロントエンド詳細
##### バックエンド詳細
```

修正後:
```markdown
# システム設計

## コンポーネント設計

* フロントエンド
	* UI コンポーネント
	* 状態管理
* バックエンド
	* API エンドポイント
	* データベース設計
```

### リスト

* アスタリスク（*）を使用します
* ハイフン（-）は使用しません
* ネストはタブを使用します
* 手順を示す場合は番号付きリストを使用できます

実装例:

修正前:
```markdown
- 項目1
  - サブ項目1-1
    - サブサブ項目1-1-1
```

修正後:
```markdown
* 項目1
	* サブ項目1-1
		* サブサブ項目1-1-1
```

### 強調表示

* 強調表示（太字・斜体）は使用しません
* 構造と階層で情報の重要度を表現します

実装例:

修正前:
```markdown
**重要**: この設定は必須です
*注意*: バックアップを取得してください
```

修正後:
```markdown
## 重要事項

* この設定は必須です

## 注意事項

* バックアップを取得してください
```

### コードブロック

* コードブロックには適切な言語指定を行います
* インラインコードは ` で囲みます

実装例:

修正前:
```markdown
以下のコマンドを実行:
```
npm install
```
```

修正後:
````markdown
以下のコマンドを実行:
```bash
npm install
```
````

インラインコード例:
```markdown
`package.json` ファイルを編集してください
```

### テキスト記述

* スペーシング
	* 英語（アルファベット）と日本語の間には半角スペースを一つ入れます
	* これにより可読性が向上します
	* 例:
		* 修正前: `LINEを使って`
		* 修正後: `LINE を使って`
		* 修正前: `GitHubリポジトリ`
		* 修正後: `GitHub リポジトリ`

## 内容規則

### コードブロックの扱い

* コードブロック内のコードは作業結果を貼り付けたものです
* 既存のコードブロックの内容は改変しません
* 作業の証跡として保持します

例:

```markdown
## 実装結果

以下のコードを実装しました:

```javascript
function example() {
  return "This code block should not be modified";
}
```

→ このコードブロックは編集せず、そのまま保持
```

### 内容の整理

* 内容は MECE（漏れなく重複なく）にまとめます
* 関連する情報は同じセクションにグループ化します
* 論理的な順序で記述します

整理の原則:

1. 重複の排除
	* 同じ情報を複数箇所に記載しない
	* 関連情報は一箇所にまとめる
2. 漏れの防止
	* 必要な情報をすべて含める
	* チェックリストで確認
3. 論理的な順序
	* 時系列順、重要度順など明確な基準で配置
	* 読み手の理解しやすい流れを意識

## コードスタイルガイドライン

Markdown ドキュメント作成時は以下を遵守してください:

* 簡潔な記述
	* 冗長な説明を避ける
	* 必要な情報のみを含める
* 一貫性の保持
	* プロジェクト全体で同じルールを適用
	* 既存ドキュメントのスタイルに準拠
* 可読性の優先
	* 階層構造を明確にする
	* 適切な空行で視認性を向上

## アンチパターン

以下のパターンは避けてください:

### H4 以降の深い見出しレベル

❌ **悪い例:**
```markdown
#### 詳細な機能説明
##### サブ機能 A
###### さらに詳細
```

✅ **良い例:**
```markdown
## 詳細な機能説明

* サブ機能 A
	* さらに詳細
```

**理由:** 深すぎる見出しレベルは構造を複雑にし、可読性を損ないます。

### ハイフンを使ったリスト

❌ **悪い例:**
```markdown
- 項目 1
- 項目 2
  - サブ項目
```

✅ **良い例:**
```markdown
* 項目 1
* 項目 2
	* サブ項目
```

**理由:** プロジェクト全体でリストマーカーを統一することで一貫性を保ちます。

### 過度な強調表示

❌ **悪い例:**
```markdown
**重要**: この設定は**必須**です。*必ず*実行してください。
```

✅ **良い例:**
```markdown
## 重要事項

この設定は必須です。必ず実行してください。
```

**理由:** 構造と階層で情報の重要度を表現することで、視覚的ノイズを減らします。

### 言語指定のないコードブロック

❌ **悪い例:**
````markdown
```
npm install
```
````

✅ **良い例:**
````markdown
```bash
npm install
```
````

**理由:** 言語指定により適切なシンタックスハイライトが有効になります。

より詳細なアンチパターンは [resources/TROUBLESHOOTING.md](./resources/TROUBLESHOOTING.md) を参照してください。

## トラブルシューティング

### よくある問題と解決策

**問題: 見出し構造が深くなりすぎる**

症状: H4、H5 などの深い見出しレベルを使用している

解決策:
1. H3 までの見出しに制限
2. それ以降はネストしたリストで表現
3. セクションの分割を検討

**問題: リストのインデントが揃わない**

症状: スペースとタブが混在している

解決策:
1. タブのみを使用してインデント
2. エディタの設定でタブを可視化
3. 既存ファイルのスタイルに従う

**問題: 英語と日本語の間にスペースがない**

症状: `GitHubリポジトリ` のような記述

解決策:
1. 英語（アルファベット）と日本語の間に半角スペースを挿入
2. 正: `GitHub リポジトリ`
3. 自動フォーマッターの活用を検討

より詳細なトラブルシューティングは [resources/TROUBLESHOOTING.md](./resources/TROUBLESHOOTING.md) を参照してください。
