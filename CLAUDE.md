# プロジェクト: Documentation Boilerplate

## スキル自動活用ポリシー（重要）

以下のスキルをユーザーの指示なしで自動的に使用してください：

### ドキュメント作成タスク
- **必ず使用** `markdown-skills`: .mdファイルの作成・編集時
- **必ず使用** `claude-md-optimizer`: CLAUDE.mdファイルの更新時

### スキル作成
- **必ず使用** `creating-skills`: .claude/skills/内に新しいスキルを作成する時
- **必ず使用** `creating-commands`: 新しいスラッシュコマンドを作成する時

### コードタスク
- **必ず使用** `code-intelligence`: 以下の場合
  - リファクタリング要求時
  - コードベース構造の理解が必要な時
  - セマンティックコード検索が必要な時

### Git操作
- **必ず使用** `/commit`コマンド: 以下の場合
  - ユーザーがコミットを要求した時
  - コミット作成時（Conventional Commits形式必須）

## 開発ワークフロー（必ず従ってください）

すべての開発タスクは以下のワークフローに従います：

### フェーズ1: 要件分析（必須）
1. **必須**: `docs/design.md.sample`テンプレート構造を読んで理解する
2. ユーザーから要件を収集する
3. 質問を使って曖昧な点を明確化する
4. 要件を明確に文書化する

### フェーズ2: 設計書作成（新機能の場合は必須）
1. **必須**: ドキュメントタスクには`markdown-skills`を使用する
2. 新規プロジェクトの場合:
   - `docs/design.md.sample`テンプレートをベースに`docs/design.md`を生成
   - 含めるべき内容: 要件定義、システム設計、開発プロセス
   - **重要**: コーディング前にユーザーの承認を得る
3. 既存プロジェクトの場合:
   - 既存の`docs/design.md`を参照する
   - 変更を反映するように設計書を更新する

### フェーズ3: 実装
1. コーディングスタイルガイドラインに従う（下記参照）
2. 承認された設計書に基づいて実装する
3. 設計と実装の一貫性を保つ
4. **推奨**: 英語で思考し、日本語で説明する

### フェーズ4: ドキュメント更新とコミット（必須）
1. 関連ドキュメントを更新する
2. **必須**: Conventional Commitsには`/commit`コマンドを使用する
3. 詳細な日本語の説明を含める

---

## 設計書ガイドライン

### 新規開発
新規開発を開始する際は、**必ず以下を実施**:
1. `docs/design.md.sample`テンプレートをベースとして使用
2. 以下を含む`docs/design.md`を自動生成:
   - Phase 1: 要件定義
   - Phase 2: システム設計
   - Phase 3: 開発プロセス
3. **重要**: コーディング前にユーザーに設計書のレビューを依頼

### 既存ソフトウェアの修正
既存ソフトウェアを修正する際は:
1. 既存の`docs/design.md`を参照
2. 設計に従ってソフトウェアを開発
3. 変更を反映するように設計書を更新

### 設計書作成プロセス
1. Phase 1: ユーザーから要件を受け取る
2. Phase 2: `docs/design.md.sample`から設計書を自動生成
3. Phase 3: ユーザーが生成された設計書をレビュー
4. Phase 4: フィードバックに基づいて設計を調整
5. Phase 5: 最終設計に基づいてコードを実装
6. **重要**: コーディング前に必ず設計書のレビューを依頼

## 利用可能なツール

### スキル（自動活用）
- `markdown-skills`: ドキュメント作成・編集（.mdファイル操作時に自動活用）
- `code-intelligence`: リファクタリング、セマンティック検索（コードタスク時に自動活用）
- `creating-skills`: スキル作成ガイド（スキル作成時に自動活用）
- `creating-commands`: コマンド作成ガイド（スラッシュコマンド作成時に自動活用）
- `claude-md-optimizer`: CLAUDE.md最適化（CLAUDE.md編集時に自動活用）

### カスタムコマンド
- `/commit`: Conventional Commits形式で日本語説明付きコミットを作成（自動的に使用）

### エージェント（Taskツール経由で使用）
- **Explore**: コードベース理解（デフォルトで"medium"の徹底度を使用）
- **Plan**: 複数ステップのタスク計画（複雑な実装に使用）

## 期待する回答形式（重要）

- **英語で思考し、日本語で説明**: 内部処理は英語で行い、詳細な日本語説明を提供
- **一貫性の維持**: 設計と実装の整合性を確保
- **詳細な説明**: 包括的な日本語ドキュメントを提供

---

## コーディングスタイル（重要）

### 一般原則
- コードベース全体で一貫性を保つ
- 言語固有のベストプラクティスに従う
- 複雑なロジックにはコメントで説明を記述

### ドキュメント標準
- すべての.mdファイル操作には`markdown-skills`を使用
- `docs/`ディレクトリのテンプレートに従う
- コード変更に合わせてドキュメントを最新に保つ

---

## Gitワークフロー（重要）

### コミット
- **必ず使用** `/commit`コマンド（Conventional Commits形式）
- フォーマット: `type(scope): description` + 詳細な日本語本文
- タイプ: feat, fix, docs, style, refactor, test, chore
- **必須**: コミット本文に日本語の説明を含める

### ワークフロー
1. 設計書に従って変更を行う
2. 関連ドキュメントを更新
3. `/commit`コマンドを使用してコミット作成
4. コミットメッセージがConventional Commitsに従っていることを確認

## 環境セットアップ

### 前提条件
- Claude Code CLIがインストールおよび設定済み
- Gitリポジトリが初期化済み
- `docs/`ディレクトリの設計テンプレートへのアクセス

### はじめに
1. `docs/design.md.sample`を確認してプロジェクト構造を理解
2. 新機能の場合: 設計書の生成を依頼
3. 既存機能の修正の場合: 既存の`docs/design.md`を参照
4. 開発ワークフローに従う（上記参照）

## 既知の規約

### 設計第一アプローチ
- **重要**: 実装前に設計書を作成・更新する必要がある
- 設計書は唯一の信頼できる情報源として機能
- すべてのコード変更は設計判断にトレースできる必要がある

### バイリンガル開発
- 内部処理: 英語（推論と技術的正確性のため）
- ユーザー向けコンテンツ: 日本語（明確なコミュニケーションのため）
- コードコメント: コンテキストに適した言語

### テンプレート駆動開発
- `docs/`ディレクトリのテンプレートを出発点として使用
- 一貫性のためにテンプレート構造を維持
- 特定の要件に基づいてテンプレートをカスタマイズ
