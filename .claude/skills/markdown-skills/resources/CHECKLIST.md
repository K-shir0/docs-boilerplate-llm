## 記述前チェックリスト

新しい Markdown ドキュメントを作成する前に確認:

* [ ] ドキュメントの目的を明確化
* [ ] 対象読者を特定
* [ ] 既存のテンプレートを確認
	* [ ] 設計書: `docs/design.md.sample`
	* [ ] プロジェクト固有のテンプレート
* [ ] 必要な情報の範囲を定義
* [ ] 類似ドキュメントのスタイルを確認

## 記述中チェックリスト

### 構造

* [ ] H1 はドキュメントタイトルとして1つのみ使用
* [ ] H2-H3 のみを使用（H4 以降は使用しない）
* [ ] 論理的な階層構造になっている
* [ ] 長文の場合は目次を追加
* [ ] セクション間に適切な空行を配置

### 見出し

* [ ] 見出しに連番を使用していない
* [ ] 見出しは簡潔で明確
* [ ] 見出しの前後に空行を配置
* [ ] 英語と日本語が混在する場合は半角スペースで区切り

### リスト

* [ ] アスタリスク（*）のみを使用
* [ ] ハイフン（-）やプラス（+）を使用していない
* [ ] タブでインデント（スペースではない）
* [ ] 番号付きリストは順序が重要な場合のみ使用
* [ ] リストのネストレベルは適切（通常は3レベルまで）

### コードブロック

* [ ] すべてのコードブロックに言語指定を追加
	* [ ] bash/sh - シェルコマンド
	* [ ] javascript/typescript - JS/TS コード
	* [ ] json - JSON データ
	* [ ] yaml - YAML 設定
	* [ ] その他適切な言語
* [ ] インラインコードは `` ` `` で囲む
* [ ] コードブロックの前後に説明を追加
* [ ] 長いコードは必要な部分のみ抜粋

### テキスト

* [ ] 英語と日本語の間に半角スペースを挿入
	* [ ] 例: `GitHub リポジトリ`
	* [ ] 例: `API エンドポイント`
* [ ] 強調表示（**太字**、*斜体*）を使用していない
* [ ] 感嘆符（!）などの感情的表現を避ける
* [ ] 簡潔で明確な表現を使用

### リンク

* [ ] すべてのリンクが有効
* [ ] 内部リンクのアンカーが正しい
* [ ] 外部リンクには説明を追加
* [ ] 相対パスは正しく設定

### 内容

* [ ] MECE（漏れなく重複なく）原則に従う
* [ ] 情報の重複を避ける
* [ ] 論理的な順序で配置
* [ ] 必要な情報がすべて含まれている
* [ ] 古い情報や不要な情報を削除

## 自動化ツール

### markdownlint

Markdown の linter ツール。記述規則の自動チェックに使用。

インストール:

```bash
npm install -g markdownlint-cli
```

使用方法:

```bash
# 単一ファイルのチェック
markdownlint README.md

# ディレクトリ全体のチェック
markdownlint docs/

# 自動修正
markdownlint --fix README.md
```

設定ファイル `.markdownlint.json`:

```json
{
	"default": true,
	"MD004": {
		"style": "asterisk"
	},
	"MD007": {
		"indent": 4
	},
	"MD013": false,
	"MD033": false,
	"MD041": false
}
```

### Prettier

コード整形ツール。Markdown の自動フォーマットに使用。

インストール:

```bash
npm install -g prettier
```

使用方法:

```bash
# 単一ファイルのフォーマット
prettier --write README.md

# ディレクトリ全体のフォーマット
prettier --write "docs/**/*.md"
```

設定ファイル `.prettierrc`:

```json
{
	"printWidth": 100,
	"tabWidth": 4,
	"useTabs": true,
	"proseWrap": "preserve"
}
```

### remark

Markdown プロセッサ。より高度な変換や検証に使用。

インストール:

```bash
npm install -g remark-cli remark-preset-lint-recommended
```

使用方法:

```bash
# チェック
remark README.md

# 自動修正
remark README.md --output
```

### VS Code 拡張機能

#### markdownlint (David Anson)

* リアルタイムで Markdown の問題を検出
* 問題箇所に波線表示
* 自動修正機能

設定例 (`.vscode/settings.json`):

```json
{
	"markdownlint.config": {
		"MD004": {
			"style": "asterisk"
		},
		"MD007": {
			"indent": 4
		}
	}
}
```

#### Markdown All in One

* 目次の自動生成
* リスト編集の補助
* キーボードショートカット
* プレビュー機能

#### Prettier - Code formatter

* 保存時の自動フォーマット

設定例:

```json
{
	"editor.formatOnSave": true,
	"[markdown]": {
		"editor.defaultFormatter": "esbenp.prettier-vscode"
	}
}
```

### pre-commit フック

Git commit 前に自動チェック。

`.git/hooks/pre-commit`:

```bash
#!/bin/sh

# Markdown ファイルのチェック
markdownlint $(git diff --cached --name-only --diff-filter=ACM | grep '\.md$')

if [ $? -ne 0 ]; then
	echo "Markdown lint エラーが検出されました。修正してください。"
	exit 1
fi
```

### GitHub Actions

CI/CD での自動チェック。

`.github/workflows/markdown-lint.yml`:

```yaml
name: Markdown Lint

on:
  push:
    paths:
      - '**.md'
  pull_request:
    paths:
      - '**.md'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run markdownlint
        uses: articulate/actions-markdownlint@v1
        with:
          config: .markdownlint.json
          files: '**/*.md'
```

## レビュー観点

Markdown ドキュメントをレビューする際の観点:

### 構造的観点

* **階層の適切性**
	* [ ] 見出しレベルは H2-H3 に制限されているか
	* [ ] 論理的な階層構造になっているか
	* [ ] セクションの粒度は適切か

* **目次の妥当性**
	* [ ] 目次が実際の構造と一致しているか
	* [ ] すべてのリンクが機能するか

### スタイル観点

* **記述規則の準拠**
	* [ ] リストはアスタリスクとタブを使用しているか
	* [ ] コードブロックに言語指定があるか
	* [ ] 英語と日本語の間にスペースがあるか
	* [ ] 強調表示を過度に使用していないか

* **一貫性**
	* [ ] ドキュメント内で用語が統一されているか
	* [ ] プロジェクト全体のスタイルと一致しているか

### 内容観点

* **正確性**
	* [ ] 技術的に正確な内容か
	* [ ] コード例が動作するか
	* [ ] リンクが有効か

* **完全性**
	* [ ] 必要な情報がすべて含まれているか
	* [ ] MECE 原則に従っているか
	* [ ] 読者が必要とする情報を提供しているか

* **明確性**
	* [ ] 説明が明確で理解しやすいか
	* [ ] 例が適切に提供されているか
	* [ ] 専門用語が適切に説明されているか

### 可読性観点

* **視覚的な読みやすさ**
	* [ ] 適切な空行が配置されているか
	* [ ] セクションが適切に分割されているか
	* [ ] コードと説明のバランスが取れているか

* **論理的な流れ**
	* [ ] 情報が論理的な順序で配置されているか
	* [ ] 読者が順序立てて理解できるか

### 保守性観点

* **更新の容易さ**
	* [ ] セクションが適切に分割されているか
	* [ ] 変更が局所的に収まるか
	* [ ] 再利用可能な構造になっているか

* **バージョン管理**
	* [ ] バージョン情報がある（必要に応じて）
	* [ ] 更新履歴が適切か（必要に応じて）

4. 自動化ツールで検証

1. レビュー観点に基づいてレビュー
2. フィードバックを具体的に
3. 記述規則への参照を含める
4. CI/CD での自動チェックを活用

### 継続的改善

1. チェックリストを定期的に見直し
2. 新しいパターンやアンチパターンを追加
3. 自動化ツールの設定を最適化
4. チーム全体で知見を共有
